# Phase 11 — MCP Removal & Unified Tool Platform

**Date:** 2026-04-28
**Status:** Spec
**Author:** Brainstorming session 2026-04-28

## 1. Problem

Nuka inherited MCP (Model Context Protocol) as the third-party extension surface. In practice the agent only needs two extension primitives — **skills** (markdown playbooks) and **tools** (callable functions with a schema). MCP brings a parallel protocol (server lifecycle, JSON-RPC framing, resources, elicitations, reconnect logic) that duplicates what the in-process tool system already does, and it is the largest single source of TUI clutter (`/mcp` slash, ElicitationDialog, Doctor checks, StatusBar segments, manifest field, monitor-mcp task kind).

This phase removes MCP entirely and replaces "third-party tool" with a single concept: a tool authored via `defineTool({...})`, registered into a single `ToolRegistry`. Tools live inside plugins; plugins may also expose a CLI binary (`bin` field) so the same capability is reachable from the terminal. Skills request tools by **capability tag** at activation time, not by name.

## 2. Goals

1. Delete every line of MCP-specific code and UI.
2. Provide a single `defineTool(...)` author surface used by both internal and plugin tools.
3. Allow a tool to declare `runtime: "in-process" | "spawn"`. Spawn tools are wrapped around a CLI binary plus a `parseOutput(stdout) => structured` adapter, letting authors lift any existing CLI into a typed tool.
4. Tag-based skill→tool lookup: skills declare `requires: string[]` (capability tags); tools declare `tags: string[]`; on skill activation the registry returns the union of tools whose tags intersect.
5. Plugins remain the single delivery mechanism; standalone external tools are simply plugins that contribute one tool plus optionally a `bin`.

## 3. Non-Goals

- No MCP compatibility shim. Existing MCP servers must be re-authored as plugin tools by their owner.
- No new sandboxing / permission gradient (existing `annotations.destructive` / `openWorld` remain the only signals).
- No TUI re-layout in this phase — that is Phase 12. This phase only removes UI hooks tied to MCP.
- No new task kinds; `monitor-mcp` is deleted, no replacement.

## 4. Architecture

### 4.1 The unified `Tool` type

Existing `Tool<I>` (`src/core/tools/types.ts`) is extended; `source` enum drops `'mcp'`.

```ts
export type ToolRuntime =
  | { kind: 'in-process' }
  | {
      kind: 'spawn'
      command: string                 // npx / absolute path / on-PATH binary
      args?: (input: unknown) => string[]
      env?: NodeJS.ProcessEnv
      parseOutput?: (stdout: string) => unknown   // default: { text: stdout }
    }

export interface Tool<I = unknown> {
  name: string
  description: string
  parameters: Record<string, unknown>
  source: 'builtin' | 'skill' | 'plugin'   // 'mcp' removed
  tags: string[]                            // NEW — capability tags for skill lookup
  runtime?: ToolRuntime                     // default { kind: 'in-process' }
  // ... existing fields unchanged: validateInput, annotations, aliases, run, needsPermission, ...
}
```

`run(input, ctx)` remains the canonical execution entry. For `runtime.kind === 'spawn'`, the default `run` is generated by `defineTool` itself: it spawns the binary, captures stdout, runs `parseOutput`, and returns the structured result. Authors who want full control still write their own `run` function.

### 4.2 `defineTool(...)` factory

New module `src/core/tools/define.ts`:

```ts
export function defineTool<I>(spec: ToolSpec<I>): Tool<I>
```

`ToolSpec<I>` is the same shape as `Tool<I>` minus the synthesised `run` for spawn-runtime. Internal tools (`read`, `write`, `bash`, ...) are migrated to `defineTool` so plugin authors and Nuka maintainers share the same author surface.

### 4.3 Skill `requires` and `Registry.queryByTags`

`SkillFrontmatter` adds `requires?: string[]`:

```yaml
---
name: deploy-helper
description: Walks the user through a deploy
requires: [git, kubectl, sql.read]
---
```

On skill activation the host calls:

```ts
ToolRegistry.queryByTags(tags: string[]): Tool[]
```

which returns all tools whose `tags` intersect `tags`.

**`requires` is additive, not a filter.** The tool set passed to the model for a skill's turn is:

```
activeTools(skill) = (tools tagged 'core' or alwaysLoad=true) ∪ queryByTags(skill.requires)
```

This means skills always retain the basic file/shell/edit toolkit and only declare *additional* capabilities they need. Skills with no `requires` (or no frontmatter `requires` field) get exactly the core set — which is current behaviour for any skill that doesn't opt in. To strip the core set entirely, a skill writes `requires: []` plus `requiresStrict: true` (escape hatch; not implemented in this phase).

### 4.4 Plugin manifest changes

`PluginManifest` schema:

- **DROP** `mcpServers` field
- **DROP** `outputStyles[].matchToolSource: 'mcp'` literal
- `tools` field semantics unchanged (relative module paths) but each module's default export is now expected to be a `Tool` produced by `defineTool` — manifest does not need a new field
- **ADD** `bin?: Record<string,string>` field (npm semantics, e.g. `{ "my-cli": "./bin/my-cli.js" }`). The schema landing happens in M3 alongside the `mcpServers` removal so manifest consumers see one breaking schema change, not two. The actual install-time symlinking of `~/.nuka/bin/<name>` lands in M4.

### 4.5 Plugin wiring (`src/core/plugin/wire.ts`)

- Delete the "MCP servers" loop; delete `mcpAdded` from the result type and all callers.
- Tool-loading path is unchanged but adds tag namespacing: `tags = [...rawTool.tags, `plugin:${plugin.name}`]` so skills can scope by plugin if desired.

### 4.6 Permission / safety

No change to the model. `annotations.destructive` / `annotations.openWorld` continue to drive permission prompts. Spawn-runtime tools inherit the same gating because they go through the same `run(input, ctx)` codepath (which is invoked under the same permission middleware).

## 7. Migration of existing internal tools

Each file in `src/core/tools/{read,write,edit,bash,glob,grep,todoWrite,htmlToMarkdown,webFetch,webSearch}.ts` is rewritten as a one-line `export default defineTool({...})`. Tag annotations chosen to keep current behaviour:

| Tool | Tags |
|---|---|
| `read` / `glob` / `grep` | `['core','fs.read']` |
| `write` / `edit` | `['core','fs.write']` |
| `bash` | `['core','shell','exec']` |
| `webFetch` / `webSearch` | `['core','net.read']` |
| `todoWrite` | `['core','session']` |
| `dispatch_agent` | `['core','agent']` |

The `'core'` tag is what makes these tools always present in `activeTools(skill)` regardless of `requires` — see §4.3.

Skills that previously got the unfiltered set continue to (no `requires`); ones that need the small set can opt in.

## 8. UI / config surface fallout

- StatusBar drops the "MCP" segment from `Segments.tsx`. `/status-bar` slash drops the `mcp` segment id.
- `src/tui/App.tsx` removes the `'elicitation'` dialog kind from its discriminated union.
- ConfigEditor drops the MCP servers panel.
- `/doctor` removes the MCP probe row.
- `/ide` keeps its existing detection (no MCP coupling there beyond the IDE-side server file paths, which are removed).
- `MessageRow` / `ToolCall` drop the `'mcp'` source branch in their renderers.

## 9. Testing strategy

- **Unit:** new tests for `defineTool` (in-process + spawn paths, parseOutput default + custom), `ToolRegistry.queryByTags`, skill `requires` parsing, manifest schema rejects `mcpServers`.
- **Integration:** `test-plans/07-tool-platform.yaml` boots a session with the example plugin, runs a skill that `requires: [git]`, asserts only `git-log` is exposed in the tool list, asserts `nuka-echo` binary is symlinked and callable from `bash`.
- **Regression:** the existing 1253 tests must all pass after migration; ones that import deleted MCP modules are deleted (the suite's MCP-only tests).

## 10. Rollout

Single phase, single PR (Phase 11). No feature flag — MCP is gone after merge. Bundle size expected to drop noticeably (15 MCP files + the SDK bridge).

## 11. Open questions

None at spec time — the Q&A loop closed all axes the author cared about. If the example plugin reveals a missing primitive (e.g. progress passthrough for spawn tools), it's a small follow-up, not a redesign.

## 12. Appendix — module dependency graph

```
defineTool ─┬─► (in-process run)
            └─► spawnRuntime ─► child_process.spawn ─► parseOutput

ToolRegistry ─► queryByTags(tags[])

skill.activation(skill) ─► registry.queryByTags(skill.requires)
                       └─► returns Tool[] for next-turn provider call

PluginLoader ─► wirePlugin
            └─► tools: register defineTool exports
            └─► slash: register slash commands
            └─► skills: parse + push
            └─► (mcpServers loop removed)
            └─► bin: symlink ~/.nuka/bin/<name> → plugin's bin entry
```

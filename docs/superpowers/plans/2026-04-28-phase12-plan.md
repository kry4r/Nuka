**Spec:** `docs/superpowers/specs/2026-04-28-phase12-tui-redesign-design.md`
**Strategy:** subagent-driven sequential development in git worktrees, merged in dependency order on `main`. Render-layer only — no agent loop / tool / session changes.

## Workstreams

### M1 — Color palette + theme refactor  (model: sonnet)

**Worktree:** `phase12-m1` off `main`.
**Goal:** introduce the 12-key semantic palette without changing visible behaviour. Every component using the old free-form keys (`accent`, `plan`, `permission`, `userMsg`, `assistantMsg`, `diffAdd`, `diffDel`, `agent.{primary,alt}`) is migrated to the new keys (`primary`, `primaryDeep`, `primarySoft`, `accentWarm`, `accentCool`, `accentInfo`, `success`, `warn`, `error`, `fg`, `fgMuted`, `fgFaint`, `bg`, `bgPanel`).

Deliverables:
1. Rewrite `src/core/theme/themes.ts`: new `ThemeColors` shape (12 semantic keys + bg + bgPanel); rewrite all five seed themes (`default-dark` rooted in avocado `#8FBF3F`; `default-light` remaps only fg/fgMuted/fgFaint/bg/bgPanel; solarized-{dark,light} and high-contrast keep their identities but expose all 12 keys). Use the dark palette table in spec §4.9 verbatim for `default-dark`.
2. Update `src/tui/theme.ts` `defaultPalette` re-export to read from active theme; add a `useColors()` thin convenience hook in `src/core/theme/context.tsx` that returns the active palette.
3. Migrate every existing color reference in `src/tui/**` to the new keys. **Do NOT bulk-replace** — the right new key depends on context:
   - Welcome braille avocado / focused frame borders / logo glyphs: `primary` (brand color).
   - dispatch_agent header (agent identity): `primary` (brand-consistent).
   - Section dividers / hint footers / metadata labels: `accentInfo`.
   - Tool-call headers / file paths in tool calls / subagent labels: `accentCool`.
   - Plan numbering: `accentInfo`.
   - Permission banner accents: `warn`.
   - User message label: `primarySoft`.
   - Diff add: `success`. Diff del: `error`.
   - Default for any other `accent` ref: `accentCool`.
   Process: enumerate every `theme.colors.*` site with grep, list each one with the proposed mapping in the PR description, batch-apply only after the list is reviewed. (One PR for the migration commit; one earlier commit can introduce the new shape.)
4. Tests: every existing render test still green; new unit test for `themes.ts` asserting each seed theme exposes all 12 keys non-empty.

Acceptance: `pnpm vitest run` green; `pnpm build` clean. No visible behaviour change in `mountApp` smoke tests.

### M2 — Layout skeleton + UIState machine  (depends on M1, model: opus)

**Worktree:** `phase12-m2` off post-M1 main.
**Goal:** the four-zone layout exists, `UIState` drives all hide rules, and existing dialogs still open/close (wired through SubmenuFrame) — but Tasks panel is empty, Status panel is the new component (StatusBar + Hud + StatusLine deleted), and slash dropdown still uses the old SlashSuggest (replaced in M5).

Deliverables:
1. Introduce `UIState` type in `src/tui/App.tsx`. Replace existing `dialog` + `slashOpen` state with a single reducer (`useReducer`) keyed by `UIState`. Esc handler always returns to `{kind:'normal'}` from any non-normal state.
2. New `src/tui/Conversation/Conversation.tsx` — wraps existing Welcome/Messages with the new frame chrome (`╭─ Conversation ─...`).
3. New `src/tui/Status/StatusPanel.tsx` — replaces `StatusBar` + `Hud` + `StatusLine`. Renders six rows per spec §4.5; respects `config.statusBar.hidden` and `config.statusBar.layout`. Auto-degrades on narrow terminals. Move legacy `statusLine` format-string interpolation into `src/tui/Status/statusLine.ts`.
4. New `src/tui/Submenu/SubmenuFrame.tsx` — common chrome for all submenus (title bar + body + footer). Distinguishes `full` vs `inline` modes via prop.
5. Wrap existing dialogs (`ModelPicker`, `SessionPicker`, `Wizard`, `StatsView`, `DoctorReport`, `MessageSelector`, `PermissionDialog`, `PluginConfigDialog`) in `SubmenuFrame`. `ConfigEditor.tsx` is left in place but the new ConfigSubmenu (M4) will replace it; until then the old one renders inside the frame.
6. Apply hide rules from spec §4.3 in `App.tsx` based on `UIState`. **Inline submenu rendering**: when `UIState` is `submenu` and `submenu.kind ∈ {permission, plugin-config}`, render the dialog component **in PromptInput's slot** (replacing it), not above it. Today's App renders both PermissionDialog and PromptInput simultaneously — that bug must not survive M2.
6a. **Welcome stays outside the Conversation frame**: when `messages.length === 0`, render `Welcome` raw at the top (not wrapped by Conversation chrome) so the centered avocado logo keeps its full canvas. The Conversation frame only applies once messages exist.
7. Focus-ring rule: pass `focused: boolean` to each frame, frames render their border with `primary` (focused) or `fgMuted` (not).
8. Delete: `src/tui/Status/Hud.tsx`, `src/tui/StatusBar/StatusBar.tsx`, `src/tui/StatusBar/Segments.tsx`, `src/tui/StatusBar/HintLine.tsx`, `src/tui/StatusLine/StatusLine.tsx`. Delete their tests under `test/tui/`.
9. Schema migration in `src/core/config/load.ts`: map old StatusBar segment ids to new ones (spec §5.1) with `Array.from(new Set(...))` dedupe. Add `statusBar.layout` field with default `'dense'`.
10. Tests:
    - `test/tui/Layout.harness.test.tsx` — asserts four zones in normal state.
    - `test/tui/Status.harness.test.tsx` — dense/compact/oneline rendering, hidden filter.
    - `test/tui/Submenu.harness.test.tsx` — opens `/config` (still old ConfigEditor), asserts hide rules; Esc returns to normal.
    - Update `test/tui/PromptInput/SlashSuggest.harness.test.tsx` to assert new "Status replaced by slash" instead of old StatusBar visibility expectations.
11. **Post-refactor verification**: `grep -rIn 'slashOpen\b' src/` must return zero hits (vestigial state after `UIState` reducer takeover). Same check for any orphan `setDialog(` calls.

Acceptance: `pnpm vitest run` green (deleted-component tests removed); `pnpm build` clean; manual smoke through harness shows four zones and dialog open/close; `grep slashOpen src/` returns zero.

### M3 — Tasks panel  (depends on M2, model: sonnet)

**Worktree:** `phase12-m3` off post-M2 main.
**Goal:** the Tasks panel is populated with Plan checklist + Subagents + Backgrounds, collapsible on `Ctrl+T`, hidden when all empty.

**Pre-flight grounding** (read before implementing):
- `src/core/tools/todoWrite.ts` exports `createTodoStore(): TodoState` — a `{ items: { title; status: 'pending'|'in_progress'|'completed' }[] }` mutable store. The `TodoWrite` tool replaces `store.items` on each call. There is **no `failed` status** on todos.
- The store is currently created and held in `cli.tsx` (where the tool factory is built) but **not** passed to `App.tsx`. M3 must thread the same `todoStore: TodoState` reference through `cli.tsx → App.tsx → TasksPanel → PlanList`.
- `src/core/tasks/types.ts` defines `TaskKind = 'local_bash' | 'local_agent'`. There is no `monitor` kind. BackgroundList renders **all** `taskManager.list()` entries.
- Subagent in-flight detection: scan `session.messages` for `tool_use` blocks named `dispatch_agent`; for each, check whether a later `tool_result` block with the same `tool_use_id` exists. Missing result → in-flight.

Deliverables:
1. New `src/tui/Tasks/TasksPanel.tsx` — composes the three sub-lists; obeys collapse state.
2. New `src/tui/Tasks/PlanList.tsx` — reads from `props.todoStore.items`; renders `✓` (completed) / `▶` (in_progress) / `☐` (pending) with index numbering. **Do not** render `✗` — TodoState has no failed status.
3. New `src/tui/Tasks/SubagentList.tsx` — scans `props.session.messages` per the algorithm above; renders `▶ running` / `✓ done` / `✗ failed` with elapsed time computed from the `tool_use` block's `ts` if present, else "—".
4. New `src/tui/Tasks/BackgroundList.tsx` — calls `props.taskManager.list()`; renders by `state` field: `▶ running`, `✓ completed`, `✗ failed`, `◉ killed`, `☐ pending`. Description from `task.description`; no detail subline (Phase 12 keeps it single-row per task).
5. Thread `todoStore` from `cli.tsx` through to `App.tsx` (new `todoStore: TodoState` prop on `AppProps`) and into `TasksPanel`.
6. Wire `Ctrl+T` in `App.tsx`: toggles between `{kind:'normal'}` and `{kind:'tasks-collapsed'}`. Update Prompt footer hint to include `Ctrl+T 任务`.
7. Empty-section hiding + total-empty frame hiding logic.
8. 12-row cap with `… +N more` per section overflow.
9. Tests:
    - `test/tui/Tasks.harness.test.tsx` — stub `todoStore` with three items; inject an in-flight `dispatch_agent` tool_use into `session.messages`; stub `taskManager.list()` returning two background tasks. Assert three sections render in order (Plan → Subagents → Backgrounds), correct icons. Toggle Ctrl+T → collapsed summary. All-empty → frame hidden.
    - Unit tests for each list component.

Acceptance: `pnpm vitest run` green; `pnpm build` clean.

### M4 — Submenu system + ConfigSubmenu  (depends on M2, model: opus)

**Worktree:** `phase12-m4` off post-M3 main (waits for M3 to keep merge linear; logically only depends on M2).
**Goal:** the Config submenu (left nav + right form) is the new `/config` UX, replacing the old ConfigEditor's "open in editor" flow. All nine category forms exist with text/password/select/toggle field types.

Deliverables:
1. New `src/tui/Submenu/config/ConfigSubmenu.tsx` — left rail (18 cols, `j/k` navigation) + right form area. Drives focus state.
2. New `src/tui/Submenu/config/Field.tsx` — primitive: label + value + edit/view modes. Field types: text, password (masked), select (single, dropdown), toggle (bool).
3. Nine category forms in `src/tui/Submenu/config/`:
    - `ProviderForm.tsx` — provider id select + base URL + api key (masked) + verify status.
    - `ModelForm.tsx` — model selector (depends on active provider).
    - `ThemeForm.tsx` — name select among five seed themes.
    - `StatusBarForm.tsx` — `layout` radio (dense/compact/oneline) + per-segment hidden checklist (`mode/model/cwd/context/cost-time/counts`).
    - `VimForm.tsx` — single toggle (`enabled`).
    - `PluginsForm.tsx` — read-only list of currently enabled plugin names + count (editing plugin enable list is out of scope, see spec §8).
    - `SkillsForm.tsx` — read-only list of registered skills (no editable fields in Phase 12).
    - `WelcomeForm.tsx` — tips list (read-only).
    - `CompactForm.tsx` — keepTurns (number) + autoThreshold (number 0-1) + contextWindow (number).
4. `s` saves via existing `src/core/config/save.ts`. Validation failure flashes the field's frame `error` for 1.5s.
5. `o` opens `~/.nuka/config.yaml` in `$EDITOR` and closes the submenu.
6. Update `src/slash/config.ts` to dispatch `{kind:'config'}` submenu instead of the old `config-editor` dialog. Delete `src/tui/dialogs/ConfigEditor.tsx`.
7. Update `App.tsx` submenu rendering switch to handle `kind:'config'`.
8. Tests:
    - `test/tui/Submenu/config/ProviderForm.test.tsx` — edit/save/cancel a field; validation error flash.
    - `test/tui/Submenu/config/StatusBarForm.test.tsx` — layout switch updates `config.statusBar.layout`; hidden checklist round-trips.
    - `test/tui/Submenu/config/ConfigSubmenu.harness.test.tsx` — `j/k` navigation between categories; right pane re-renders.

Acceptance: `pnpm vitest run` green; `pnpm build` clean; manual smoke: `/config` opens new submenu, switching `StatusBar.layout` to `oneline` is visible after save.

### M5 — Slash card (grouped + arg-hint)  (depends on M2, model: sonnet)

**Worktree:** `phase12-m5` off post-M4 main.
**Goal:** the slash dropdown is replaced by the new SlashCard that takes over the Status slot when open, groups commands by source, and morphs into an arg-hint card when the user types space.

Deliverables:
1. Extend `SlashCommand` interface in `src/slash/types.ts` with optional `source: 'builtin' | 'plugin' | 'skill'`, `shortcut: string`, `args`, `examples` (additive, non-breaking).
2. Populate the new fields on built-in commands that benefit (`/model`, `/skill`, `/config`, `/tasks`, `/sessions`, `/help`, `/stats`, `/doctor`). Plugin and skill commands tag their `source` automatically at registration time (in `src/core/plugin/wire.ts` and skill-as-slash registration).
3. New `src/tui/SlashCard/SlashCard.tsx` — top-level container; switches between `CommandList` and `ArgHint` based on whether value contains space.
4. New `src/tui/SlashCard/CommandList.tsx` — grouped list (builtins / plugins / skills); selected row highlighted with `primaryDeep` background; pagination with ellipsis hints.
5. New `src/tui/SlashCard/ArgHint.tsx` — renders Usage/Args/Examples for fully-populated commands; degenerate single-line `Usage: /<name>` for sparse ones.
6. Update `App.tsx`: when `UIState.kind === 'slash'`, hide Tasks, raise Prompt to where Tasks was, render SlashCard in the Status slot. PromptInput's `onSlashActiveChange` callback drives the state transition.
6a. **PromptInput trigger logic change**: today (`PromptInput.tsx` ~L42-43) slash candidates are dropped the moment a space is typed (`if (props.value.includes(' ')) return []`). New behaviour: as long as `value.startsWith('/')` (regardless of space), `slashActive` stays true. The `slash:list` ↔ `slash:arg-hint` distinction lives **inside** `SlashCard` (it switches sub-component based on whether the value contains a space). Update `PromptInput.tsx` accordingly. Keep `Tab` accept behaviour (in `slash:list` mode only; in `arg-hint` mode `Tab` is a no-op).
6b. **Footer hint refresh**: drop "Tab 接受" from the prompt footer hint while in `slash:arg-hint` mode (since Tab does nothing there).
7. Delete `src/tui/PromptInput/SlashSuggest.tsx` and its test (replaced by SlashCard tests).
8. Tests:
    - `test/tui/SlashCard.harness.test.tsx` — types `/`, asserts grouping (`builtins (N)`, `skills (M)`); types `/model `, asserts arg-hint card.
    - Unit test for `CommandList` grouping + selection.
    - Unit test for `ArgHint` full vs degenerate render.

Acceptance: `pnpm vitest run` green; `pnpm build` clean; manual smoke: `/` shows grouped list; `/model ` shows arg-hint card; Esc closes both.

## M1 — Harness core

### 9.1 Plan parser + validator
**Files:** `src/core/testing/plan.ts`, `test/core/testing/plan.test.ts`.

`parsePlan(yamlText: string): Plan` validates schema and returns a typed plan or throws `PlanError {line, column, message}`.

Acceptance:
1. Valid YAML with all step kinds parses.
2. Unknown step kind → throws.
3. Missing `name` → throws.
4. `mockResponses` with malformed delta → throws.

### 9.2.a Mock provider + keystroke constants
**Files:** `src/core/testing/mockProvider.ts`, `src/core/testing/keystrokes.ts`. Tests: `test/core/testing/{mockProvider,keystrokes}.test.ts`.

Acceptance:
1. `MockProvider` implements LLMProvider; yields scripted deltas in order.
2. `keystroke('UP')` returns `\u001B[A`; `'ENTER'` returns `\r`; etc.

### 9.2.b Inkharness wrapper
**Files:** `src/tui/testing/harness.ts`. Tests: `test/tui/testing/harness.test.tsx`.

`mountApp({config, mocks, target?})` returns `{stdin, frames(), unmount, waitFor(matcher, timeoutMs?)}`. The `target` chooses what to render: `'app'` mounts `<App {...mocks}>`, `'wizard'` mounts `<Wizard>`. Use ink-testing-library `render` under the hood.

Acceptance:
1. Mounting `'wizard'`, sending Enter → state transitions visible in `frames()`.
2. `waitFor({contains: 'X'}, 50ms)` resolves once frame contains X; rejects on timeout.

### 9.3 Runner + assertions
**Files:** `src/core/testing/runner.ts`, `src/core/testing/assertions.ts`. Tests: `test/core/testing/runner.test.ts` + `assertions.test.ts`.

`runPlan(plan, opts)` walks `steps[]`, dispatches by step kind, accumulates `frames[]` and `failures[]`. Snapshot step compares to `test-plans/__snapshots__/<name>.txt`; if `updateSnapshots`, overwrites.

Acceptance:
1. A 5-step plan with all step kinds runs end-to-end; frames captured.
2. A failing `assert` produces a `RunResult{ok:false, steps:[...{ok:false, message:...}]}`.
3. Snapshot mismatch produces a diff in the failure message; `updateSnapshots:true` rewrites the file.

### 9.4 Vitest binding helper
**Files:** `src/core/testing/vitest.ts`. Tests: `test/core/testing/vitest.test.ts`.

`expectPlanToPass(yamlText)` → vitest-friendly assertion that wraps `runPlan`.

Acceptance:
1. A passing plan runs in vitest with no extra plumbing.
2. A failing plan surfaces the failure message via `expect.fail`.

## M2 — CLI mode + sample plans

### 9.5 `nuka --test-plan` subcommand
**Files:** `src/cli.tsx` (new argv branch). Tests: `test/cli/testPlan.test.ts`.

Args: `--test-plan <path> [--update-snapshots] [--reporter=tap|json|pretty]`. Defaults to `pretty`.

Acceptance:
1. Running with a passing plan exits 0 and prints a checkmark per step.
2. Running with a failing plan exits 1 with a red diff section.
3. Invalid YAML exits 2 with a clear parse error.

### 9.6 Five sample plans + integration tests
**Files:** `test-plans/01-offline-boot.yaml`, `02-onboarding.yaml`, `03-theme-switch.yaml`, `04-stats-view.yaml`, `05-plan-mode-lockout.yaml`. Tests: `test/integration/samplePlans.test.ts` runs each via `runPlan` and asserts all pass.

Plans cover:
1. Offline boot — banner present, prompt accepts `/help`.
2. Onboarding — type `/init` (or render the wizard directly), pick anthropic, enter key, pick model, see "saved".
3. Theme switch — type `/theme default-light`, observe HUD accent change in next frame.
4. Stats view — type `/stats`, see Overview tab, press `Tab`, see Models tab, press `Esc`.
5. Plan-mode lockout — type `/plan on`, send a message that would trigger a Write tool, observe the agent gets a `tool_result {isError:true, output:/plan mode/}`.

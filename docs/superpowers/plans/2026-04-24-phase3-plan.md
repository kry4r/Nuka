## §A · L3a — ToolRegistry conflict resolution

**Why first:** `ToolRegistry.register` currently throws on duplicate name. Phase 3 spec acceptance for L3: *"Duplicate names are logged; the earlier registration wins."* Loosening this up front avoids throwing during L1c/L2b wiring.

**Files touched:**
- `src/core/tools/registry.ts` — change `register` behavior.
- `test/core/tools/registry.test.ts` — cover new behavior.

**Contract change:**
```ts
class ToolRegistry {
  register(tool: Tool): { registered: boolean; reason?: string }
  // registered: true if accepted, false if dropped (duplicate)
  // on duplicate: console.warn(`tool '${name}' already registered from ${existing.source}; skipping ${tool.source}`)
  bySource(source: Tool['source']): Tool[]
}
```

**Acceptance:**
1. Registering two tools with the same name — the second `register` returns `{ registered: false, reason: 'duplicate' }` and logs a warning; `find(name).source` equals the first one.
2. `bySource('builtin')` returns only builtin tools; filter covers all four sources.
3. Existing tests that expect `register` to *throw* on duplicates are updated to expect the new return shape instead.

## §C · L1b — MCP client + manager + tool adapter

**New dependency:** `npm install @modelcontextprotocol/sdk`. Verify it externalizes cleanly (already a runtime dep per `scripts/build.mjs`).

**Files created:**
- `src/core/mcp/client.ts` — thin wrapper:
  ```ts
  export class McpClient {
    constructor(opts: { name: string; config: McpServerConfig; onStatusChange?: (s: McpConnectionStatus) => void })
    connect(signal?: AbortSignal): Promise<void>   // picks transport by config.type; starts SDK client; calls onStatusChange
    listTools(): Promise<McpToolDescriptor[]>      // cached after first call; invalidated on reconnect / tools/list_changed notification
    listResources(): Promise<McpResourceDescriptor[]>
    callTool(name: string, input: unknown, signal?: AbortSignal): Promise<{ output: string; isError: boolean }>
    // converts content blocks (text/image/resource_link) into a single string output; binary blobs summarized
    readResource(uri: string, signal?: AbortSignal): Promise<{ output: string; isError: boolean }>
    close(): Promise<void>
    readonly status: McpConnectionStatus
    readonly name: string
  }
  ```
  Transport:
  - `stdio` → `StdioClientTransport({ command, args, env })`
  - `http` → `StreamableHTTPClientTransport(new URL(url), { requestInit: { headers } })`
- `src/core/mcp/manager.ts`:
  ```ts
  export class McpManager {
    constructor(opts: { servers: Record<string, McpServerConfig> })
    startAll(): Promise<void>          // connects every server in parallel; failures do NOT throw — recorded in status
    status(): Array<{ name: string; status: McpConnectionStatus }>
    listClients(): McpClient[]
    findClient(name: string): McpClient | undefined
    closeAll(): Promise<void>
    onChange(listener: () => void): () => void   // returns unsubscriber; fires on any client status change
  }
  ```
- `src/core/mcp/toolAdapter.ts`:
  ```ts
  export function mcpToolsFor(client: McpClient): Promise<Tool[]>
  // returns one Nuka Tool per MCP tool: name = buildMcpToolName(client.name, t.name),
  // description copies MCP tool's, parameters copies MCP tool's inputSchema,
  // source: 'mcp', needsPermission: () => 'exec',
  // run delegates to client.callTool (using raw MCP tool name, not namespaced)
  ```

**Tests:**

**Acceptance:**
1. Given a mocked SDK client, `McpClient.connect` + `listTools` + `callTool` round-trip text results to strings; image blobs become a `[binary: <mime> len=N]` summary line.
2. `McpManager.startAll` keeps running when one server errors; status reflects both outcomes.

  import { z } from 'zod'

  export const PluginManifestSchema = z.object({
    name: z
      .string()
      .min(1)
      .regex(/^[a-z0-9][a-z0-9-]*$/i, 'kebab-case letters/numbers only'),
    version: z.string().optional(),
    description: z.string().optional(),
    tools: z.array(z.string()).optional(),         // relative import paths to modules that default-export a Tool

    mcpServers: z.record(z.string(), McpServerConfigSchema).optional(),
  })
  export type PluginManifest = z.infer<typeof PluginManifestSchema>

    manifest: PluginManifest
    rootDir: string   // absolute path to plugin dir
  }
  ```

  // returns plugins in filesystem-sorted order (stable).
  ```

## §F · L2b — Plugin contribution wiring

**Files created:**
- `src/core/plugin/wire.ts`:
  ```ts
  export async function wirePlugin(
    plugin: LoadedPlugin,
    deps: {
      tools: ToolRegistry
      slash: SlashRegistry
      skills: Skill[]            // append-to array (mutation allowed per Phase-2 pattern)
      mcpServers: Record<string, McpServerConfig>  // append-to map
    },
  ): Promise<{
    toolsAdded: number
    slashAdded: number
    skillsAdded: number
    mcpAdded: number
  }>
  ```
  Behavior:
  - Tools: `await import(path)` each declared tool module, take default export; wrap its `.name` as `plugin__${plugin.manifest.name}__${rawName}`; set `source: 'plugin'`; call `tools.register` (conflicts skipped per L3a).
  - Slash commands: same import pattern; rename to `${plugin.manifest.name}:${rawName}`; call `slash.register`.
  - Skills: for each path/glob, read the md file, `parseSkill(...)`, add to skills list if valid.
  - MCP servers: for each entry `[key, value]`, assign into `mcpServers[key]` unless the key already exists (log skip).

**Files modified:**
- `src/cli.tsx`:
  - After `loadSkills` and before constructing McpManager, call `await loadPlugins({ home })` and for each plugin call `wirePlugin(...)`. Order: plugins wire AFTER skills load (so plugins can add more skills) and BEFORE McpManager (so plugins can contribute mcp servers into the map).

**Tests:**
- `test/core/plugin/wire.test.ts` — using a temp plugin dir with a tool module, slash module, skill md, and one mcpServers entry: `wirePlugin` returns `{ 1, 1, 1, 1 }` and the registries contain the namespaced entries.

**Acceptance:**
1. A sample plugin `~/.nuka/plugins/demo/` with `plugin.yaml` declaring one tool module registers `plugin__demo__<raw>` in the tool registry.
2. Duplicate tool name across builtin + plugin: builtin survives, plugin tool is logged and skipped.

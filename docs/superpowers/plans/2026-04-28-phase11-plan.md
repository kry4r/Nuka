# Phase 11 — Implementation Plan

**Spec:** `docs/superpowers/specs/2026-04-28-phase11-mcp-removal-tool-platform-design.md`
**Strategy:** subagent-driven parallel development in git worktrees, merged in dependency order on `main`.

## Workstreams

### M1 — Tool platform foundation  (model: opus)

**Worktree:** `phase11-m1` off `main`.
**Goal:** introduce the new author surface without yet breaking MCP. Everything compiles + every existing test passes.

Deliverables:
1. `src/core/tools/define.ts`: `defineTool<I>(spec)` factory.
2. `src/core/tools/spawnRuntime.ts`: helper that, given `runtime.kind === 'spawn'`, returns an `async run(input, ctx)` that spawns the binary, captures stdout (with abort + `signal` plumbing), runs the `parseOutput` adapter (default: `{ text: stdout }`), maps stderr+exitcode to `ToolResult.isError`.
3. Extend `Tool<I>` in `src/core/tools/types.ts` with `tags: string[]` (required, default to `[]`) and optional `runtime`. **Keep `'mcp'` in the source enum for now** — M3 deletes it.
4. Extend `ToolRegistry` (`src/core/tools/registry.ts`) with `queryByTags(tags: string[]): Tool[]` returning tools whose `tags` intersect.
5. Extend skill frontmatter (`src/core/skill/types.ts`, `src/core/skill/loader.ts`) with optional `requires?: string[]`.
6. New `src/core/skill/activation.ts`: `activeToolsFor(skill, registry): Tool[]` — returns `registry.queryByTags(skill.requires)` when `requires` is non-empty, else the full registry.
7. Migrate every internal tool in `src/core/tools/{read,write,edit,bash,glob,grep,todoWrite,htmlToMarkdown,webFetch,webSearch}.ts` to `defineTool({...})` with the tag table from spec §7.
8. Tests: new unit tests for define / spawnRuntime / queryByTags / skill `requires` parsing / activation. Existing tests untouched and green.

Acceptance: `pnpm vitest run` green; `pnpm build` clean.

### M2 — Internal-tool host wiring & skill activation hook  (depends on M1, model: sonnet)

**Worktree:** `phase11-m2` off post-M1 main.
**Goal:** wire `activeToolsFor` into the agent loop so `requires` actually narrows the tool set the model sees.

Deliverables:
1. Locate the call site that builds the tool list passed to the provider for the next turn (`src/core/agents/toolFilter.ts` or main loop). Insert a call to `activeToolsFor(activeSkill, registry)` when a skill is active; fall back to current behaviour otherwise.
2. Add a regression test: registers two tools (`a` tags=`[x]`, `b` tags=`[y]`), starts a skill with `requires:[x]`, asserts the provider call list contains `a` only.
3. Add a `test-plans/07-tool-platform.yaml` plan that drives the same flow through the harness end-to-end.

Acceptance: 1254+ tests green (1 new); harness plan passes via `nuka --test-plan test-plans/07-tool-platform.yaml`.

### M4 — Plugin-author surface: example plugin + bin support  (depends on M3, model: sonnet)

**Worktree:** `phase11-m4` off post-M3 main.
**Goal:** prove the new surface works end-to-end with a real, runnable example, and add the missing `bin` linking on install.

Deliverables:
1. `src/core/plugin/manifest.ts`: add optional `bin?: Record<string,string>` (path entries relative to plugin root); install path symlinks into `~/.nuka/bin/<name>` (best-effort on Windows: copy stub).
2. `src/core/plugin/install.ts`: implement bin linking on install, unlink on uninstall.
3. `examples/plugin-cli-tool/`:
   - `plugin.json` declaring 2 tools + 1 bin
   - `tools/echo.ts` — in-process tool, `tags:["util"]`, returns the input text upper-cased
   - `tools/git-log.ts` — spawn tool wrapping `git log --oneline -n 5`, with `parseOutput` returning `{commits: string[]}`, `tags:["git","vcs.read"]`
   - `bin/nuka-echo` — Node script that prints `argv[2]` upper-cased (proves bin linking works without depending on the in-process tool)
   - `skills/deploy-helper.md` with `requires:["git"]` to demonstrate skill→tool narrowing
4. `test-plans/08-example-plugin.yaml`: installs the example plugin via `nuka plugin install`, runs the deploy-helper skill, asserts the `git-log` tool fires, asserts `nuka-echo` binary is callable through `bash`.
5. `README.md` + `README.zh-CN.md`: replace the MCP section with a "Building a plugin tool" section pointing to the example.

Acceptance: harness plan 08 passes; bin symlink exists at the expected path on Linux/macOS; bundle smaller than pre-Phase-11 (sanity check, expected drop ≥ 30 KB).
